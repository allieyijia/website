Context

The Problem

UPS recently migrated their infrastructure to Red Hat OpenShift — a platform that runs containerized workloads across clusters of CPU, memory, and DASD storage. But OpenShift didn't give engineers the planning tools to go with it. Every time a team wanted to add a new workload, change cluster configurations, or prepare for growth, they had to do the analysis manually: spreadsheets, tribal knowledge, redundant individual calculations.

The problem wasn't showing cluster data. UPS already had data. The problem was using it — turning raw utilization metrics into confident decisions about whether a proposed change would fit, where the bottlenecks were, and how the clusters would perform in two years.

I joined a 5-person Agile/Scrum team as the sole Frontend UX Engineer to design and build a web-based capacity analytics platform from scratch. My teammates: Mithil (PM), Lizeth (Associate PM), Junaid (Cloud/Systems), and Abhinav (Backend).

Research

Understanding the Users

I authored Chapter 3 of the final paper — stakeholder mapping, requirements gathering, and project scope — and developed two personas to anchor design decisions across all five sprints.

Derek Infrastructure Engineer

Day-to-day cluster monitoring, responding to alerts, executing capacity changes. He needs fast answers: "Does this workload fit? What's my DASD headroom?" He has no tolerance for charts that make simple questions complicated.

Priya Platform Lead

Quarterly planning cycles, budget justification, executive reporting. She needs to model scenarios — "What if we move 20% of workloads to the new cluster?" — without touching a spreadsheet or waiting on a manual analysis.

Both personas pointed to the same core insight: analytical depth had to feel effortless. Tasks that took hours needed to feel like a two-minute conversation with the data.

Design Process

Wireframes → Figma → Blazor

I started in low-fidelity: sketched flows, mapped information hierarchy, and identified the four core modules the platform needed. Early team reviews shaped which views needed their own page vs. which could share layout. From there I moved into hi-fi Figma prototypes, establishing the visual language before writing a line of frontend code.

For the visual design, I chose a dark UPS-branded theme. Engineers working long monitoring shifts benefit from reduced eye strain, and high-contrast data visualizations (ApexCharts in gold and amber against dark backgrounds) read far more clearly than light-mode alternatives. Every color decision tied back to the UPS brand colorway.

Then I built it. Self-teaching Blazor/.NET 10 was part of the scope — there was no template to work from. I built reusable Razor components for summary cards, data tables, scenario inputs, and charts, and integrated Microsoft Entra ID for role-based authentication across all routes. The jump from "designing the interface" to "being the person who also ships it" changed how I design: every decision became more concrete, more grounded in what's actually buildable.

Feature 1

Fleet Overview & Growth Projection

The home page gives engineers a real-time snapshot of every cluster in the environment: health status (Healthy / Warning / Critical based on utilization thresholds), current CPU, RAM, and DASD utilization, and node counts. Selecting a cluster reveals a detailed view with a 3-year projection line chart — interpolated to the exact crossing point, not rounded to the nearest year — so engineers know precisely when a resource will be exhausted.

Fleet Overview: real-time cluster health grid (left) and detailed 3-year utilization projection (right). The "Add Capacity" and "Remove Workload" buttons drop users directly into the relevant planning module with the selected cluster pre-loaded.

Feature 2

Addition Modeling

Engineers can specify workloads and nodes to add — CPU cores, memory GB, DASD, replica counts — and see the projected impact before touching production. Remaining headroom updates dynamically as inputs change, with color-coded status (Healthy / Watch / Critical). The Growth Projection section lets users customize the annual rate and duration, then generate a scenario-specific timeline.

Left: the input configuration. Right: results — Before vs. After utilization table, Growth Projection Timeline with bottleneck detection, and overall fit status badges.

One design decision I'm proud of: the results panel appears alongside the inputs rather than replacing them. Engineers can tweak parameters and re-run without losing context — a pattern that came directly from watching how the persona of "Derek" would actually use this tool under time pressure.

Feature 3

Removal Analytics

The flip side of addition: engineers check workloads off a list to simulate decommissions, and the platform instantly recalculates reclaimed CPU, memory, and storage, updates utilization deltas, and identifies the remaining bottleneck. CPU overcommit ratio is tracked separately from memory and storage — because in OpenShift, CPU throttles while memory kills, and the UI surfaces that distinction clearly.

Removal Analytics: engineers check workloads to remove; reclaimed resources, utilization Before vs. After, and bottleneck status update in real time. The Selection Summary at bottom gives a plain-language read of the scenario.

Feature 4

What-If Analysis

The most complex view and the one that required the most design iteration: a two-step, multi-cluster scenario modeling system. Engineers configure additions and removals across multiple clusters simultaneously (with an "Apply to All" option for identical changes), then run analysis to get grouped utilization charts, baseline vs. scenario delta tables, projected exhaustion timelines, and a multi-cluster comparison view. The two-step workflow — Configure → Analyze — was a deliberate choice to separate planning inputs from result interpretation so neither muddied the other.

What-If Analysis results: grouped utilization by stage (Baseline → Post-Removal → Applied Scenario), side-by-side Requested vs. Remaining, delta comparison table, and projected utilization over time. This is the platform's most powerful view — and the one UPS sponsors engaged with most during demos.

Reflection

What I Learned

The biggest lesson from this project: constraints are design inputs. The data model, the API response structure, the Blazor component lifecycle — every technical reality shaped what was possible to design. Learning to work within those constraints, rather than around them, made me a sharper designer and a much better collaborator with backend engineers.

I also learned what it actually takes to build software that professionals can depend on. Working with real UPS sponsors, a real codebase, real authentication, and real infrastructure concepts was a different experience than any coursework. The feedback session with actual infrastructure engineers near the end of the semester revealed layout choices that made more sense to developers than to end users — and that feedback made the final product meaningfully better. I'd bring users in earlier next time.

"The hardest UX problems aren't about making things pretty — they're about making the right trade-off between what users need and what the system can do."
